Every automation, whether no-code or AI-powered, ultimately works by calling APIs. Understanding how APIs work means you can connect any two services, even ones that no-code tools don't support yet.

The API layer is the foundation of the entire automation stack. No-code platforms like n8n and Make are visual wrappers around API calls. AI agents use tool calling to invoke APIs on your behalf. Even MCP servers are structured API interfaces. When you understand REST, authentication, and webhooks at a fundamental level, you can debug any broken automation, build integrations that no pre-built connector covers, and reason about the data flowing through your pipelines. This module provides that foundation.

            <p>REST APIs use HTTP. the same protocol your browser uses. Every API call has four parts:</p>
            <ul>

const response = await fetch(

);
const data = await response.json();
console.log(\`Found \${data.jobs.length} open positions\`);`}</code></pre>

        <AccordionTrigger className="text-lg text-left">Authentication: API Keys, OAuth, and Tokens</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Most APIs require authentication. The three common patterns:</p>
            <p><strong>1. API Keys</strong>. Simple, static tokens. Include in headers or query params.</p>
            <pre><code>{`// Telegram Bot API. send a message to a channel
const TELEGRAM_TOKEN = process.env.TELEGRAM_BOT_TOKEN;
const CHANNEL_ID = process.env.TELEGRAM_CHANNEL_ID;

await fetch(\`https://api.telegram.org/bot\${TELEGRAM_TOKEN}/sendMessage\`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    chat_id: CHANNEL_ID,
    text: "5 new Web3 jobs posted today!",
    parse_mode: "HTML",
  }),
});`}</code></pre>
            <p><strong>2. Bearer Tokens</strong>. Short-lived tokens, usually from OAuth flows.</p>
            <pre><code>{`// Using a bearer token with the OpenAI API
await fetch("https://api.openai.com/v1/chat/completions", {
  method: "POST",
  headers: {
    "Authorization": \`Bearer \${process.env.OPENAI_API_KEY}\`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    model: "gpt-4o-mini",
    messages: [{ role: "user", content: "Summarize these job listings..." }],
  }),
});`}</code></pre>
            <p><strong>3. OAuth 2.0</strong>. For accessing user data (Google Sheets, Twitter, LinkedIn). Involves redirect flows and token refresh. No-code tools handle this for you; for code, libraries like <code>googleapis</code> simplify it.</p>
            <p><strong>Golden rule:</strong> Never hardcode secrets. Always use environment variables (<code>process.env.API_KEY</code>).</p>

    <AccordionItem value="item-3">
        <AccordionTrigger className="text-lg text-left">Webhooks: Real-Time Event Triggers</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>APIs let you <em>pull</em> data on demand. Webhooks let services <em>push</em> data to you when something happens. They're the "trigger" in most automations.</p>
            <p><strong>How webhooks work:</strong></p>
            <ol>
                <li>You register a URL with a service: "When a new payment arrives, POST to <code>https://myapp.com/api/webhook/stripe</code>"</li>
                <li>When the event happens, the service sends an HTTP POST to your URL with event data</li>
                <li>Your server processes the event and takes action</li>
            </ol>
            <p><strong>Real-world example: Auto-posting new jobs to social media</strong></p>
            <pre><code>{`// Next.js API route handling a webhook from your job database
export async function POST(request: Request) {
  const { job } = await request.json();
  
  // Format the job for Telegram
  const message = \`<b>\${job.title}</b>\\n\` +
    \`\${job.company}\\n\` +
    \`\${job.location}\\n\\n\` +
    \`Apply: \${job.url}\`;
  
  // Post to Telegram channel
  await fetch(\`https://api.telegram.org/bot\${BOT_TOKEN}/sendMessage\`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      chat_id: CHANNEL_ID,
      text: message,
      parse_mode: "HTML",
    }),
  });
  
  return Response.json({ success: true });
}`}</code></pre>
            <p>This pattern powers automated Telegram channels, Discord bots, email newsletters, and social media feeds. new data arrives via webhook, gets formatted, and broadcasts instantly.</p>

        <AccordionTrigger className="text-lg text-left">Error Handling Patterns for API Automation</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>APIs fail. Networks timeout. Services go down. Your automation needs to handle all of this gracefully:</p>
            <p><strong>Pattern 1: Retry with backoff.</strong> If a request fails with a 5xx error or timeout, wait and try again. Increase the wait time with each retry to avoid hammering a struggling server:</p>
            <pre><code>{`async function fetchWithRetry(url, options = {}, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      const response = await fetch(url, {
        ...options,
        signal: AbortSignal.timeout(10000), // 10s timeout
      });
      
      if (response.status === 429) {
        // Rate limited: respect Retry-After header
        const retryAfter = response.headers.get('Retry-After');
        const delay = retryAfter ? parseInt(retryAfter) * 1000 : Math.pow(2, attempt) * 1000;
        console.warn(\`Rate limited. Waiting \${delay}ms...\`);
        await new Promise(r => setTimeout(r, delay));
        continue;
      }
      
      if (!response.ok) throw new Error(\`HTTP \${response.status}\`);
      return await response.json();
    } catch (error) {
      if (attempt === maxRetries - 1) {
        console.error(\`Failed after \${maxRetries} attempts: \${url}\`);
        return null; // Return null instead of throwing. let the pipeline continue.
      }
      await new Promise(r => setTimeout(r, 1000 * Math.pow(2, attempt)));
    }
  }
}`}</code></pre>
            <p><strong>Pattern 2: Graceful degradation.</strong> If one API source fails, don't kill the entire pipeline. Use <code>Promise.allSettled</code> to run all fetches in parallel and collect whatever succeeds:</p>
            <pre><code>{`const results = await Promise.allSettled([
  fetchJobs('greenhouse', slugs),
  fetchJobs('lever', slugs),
  fetchJobs('ashby', slugs),
]);

const succeeded = results.filter(r => r.status === 'fulfilled');
const failed = results.filter(r => r.status === 'rejected');
console.log(\`\${succeeded.length} sources OK, \${failed.length} failed\`);`}</code></pre>
            <p><strong>Pattern 3: Dead letter queue.</strong> When individual items fail processing, don't drop them. Store them for manual review or re-processing later.</p>
        </AccordionContent>

    <AccordionItem value="item-5">
        <AccordionTrigger className="text-lg text-left">Building a Multi-Source Data Aggregator</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>The most valuable API automations aggregate data from multiple sources into a single, normalized format. Here's the complete pattern:</p>
            <pre><code>{`// Step 1: Define your sources with a common interface
const sources = [
  {
    name: 'Greenhouse',
    fetchFn: async (slug) => {
      const res = await fetchWithRetry(
        \`https://boards-api.greenhouse.io/v1/boards/\${slug}/jobs\`
      );
      return res?.jobs?.map(j => ({
        title: j.title,
        company: slug,
        location: j.location?.name || 'Remote',
        url: j.absolute_url,
        source: 'greenhouse',
      })) || [];
    },
    slugs: ['consensys', 'uniswap', 'chainlink'],
  },
  {
    name: 'Lever',
    fetchFn: async (slug) => {
      const res = await fetchWithRetry(
        \`https://api.lever.co/v0/postings/\${slug}?mode=json\`
      );
      return res?.map(j => ({
        title: j.text,
        company: slug,
        location: j.categories?.location || 'Remote',
        url: j.hostedUrl,
        source: 'lever',
      })) || [];
    },
    slugs: ['figma', 'notion'],
  },
];

// Step 2: Fetch all sources in parallel
const allJobs = [];
for (const source of sources) {
  const results = await Promise.allSettled(
    source.slugs.map(slug => source.fetchFn(slug))
  );
  for (const result of results) {
    if (result.status === 'fulfilled') allJobs.push(...result.value);
  }
}

// Step 3: Deduplicate by URL
const unique = [...new Map(allJobs.map(j => [j.url, j])).values()];
console.log(\`Aggregated \${unique.length} unique jobs from \${sources.length} sources\`);`}</code></pre>
            <p>This exact pattern powers job boards, price comparison engines, news aggregators, and competitive intelligence tools. Once you understand it, you can aggregate anything from any set of APIs.</p>
        </AccordionContent>

    <AccordionItem value="item-6">
        <AccordionTrigger className="text-lg text-left">Pagination & Rate Limits</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>When automating API calls, two challenges always appear:</p>
            <p><strong>Pagination:</strong> APIs don't return all data at once. They paginate. 20 items per page, 100 per page, etc. Your automation must loop through all pages:</p>
            <pre><code>{`// Fetch all jobs across all pages
let allJobs = [];
let page = 1;
let hasMore = true;

while (hasMore) {
  const response = await fetch(
    \`https://api.example.com/jobs?page=\${page}&per_page=100\`
  );
  const data = await response.json();
  allJobs.push(...data.jobs);
  hasMore = data.jobs.length === 100; // If less than 100, we're done
  page++;
  await new Promise(r => setTimeout(r, 200)); // Respect rate limits
}`}</code></pre>
            <p><strong>Rate Limits:</strong> APIs limit how many requests you can make per minute/hour. Always:</p>
            <ul>
                <li>Add delays between requests (<code>setTimeout</code>)</li>
                <li>Check for <code>429 Too Many Requests</code> responses and back off</li>
                <li>Cache responses when possible to avoid redundant calls</li>
                <li>Use batch endpoints when available</li>
            </ul>
        </AccordionContent>
