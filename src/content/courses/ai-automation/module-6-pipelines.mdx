Individual automations are useful. Pipelines that chain multiple automations together are transformative. This module teaches you to build complete, end-to-end systems using everything you've learned.

<Accordion type="single" collapsible className="w-full mt-4">
    <AccordionItem value="item-1">
        <AccordionTrigger className="text-lg text-left">Pipeline Architecture: Trigger → Fetch → Transform → Act → Report</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Every automation pipeline follows the same five-stage pattern:</p>
            <ol>
                <li><strong>Trigger:</strong> What starts the pipeline. a cron schedule, a webhook, a database change, or a manual command.</li>
                <li><strong>Fetch:</strong> Gather the input data. call APIs, read databases, scrape websites, check MCP resources.</li>
                <li><strong>Transform:</strong> Process the data. filter, deduplicate, enrich with AI, normalize formats, calculate metrics.</li>
                <li><strong>Act:</strong> Do something with the processed data. write to a database, send messages, publish content, update spreadsheets.</li>
                <li><strong>Report:</strong> Log results, send notifications, update dashboards, handle errors.</li>
            </ol>
            <p>The key to reliable pipelines is making each stage independent and idempotent. if stage 3 fails, you can re-run it without duplicating the work from stages 1-2.</p>
        </AccordionContent>
    </AccordionItem>

            <p>A real-world pipeline that aggregates job listings from 50+ company boards and broadcasts them daily:</p>
            <p><strong>Stage 1. Trigger:</strong> GitHub Actions cron runs at 6am UTC daily.</p>
            <p><strong>Stage 2. Fetch:</strong></p>

const sources = [
  // Greenhouse boards
  ...greenhouseSlugs.map(slug => 
    fetch(\`https://boards-api.greenhouse.io/v1/boards/\${slug}/jobs\`)

  ),
  // Ashby boards

const results = await Promise.allSettled(sources);`}</code></pre>

                <li>POST to Telegram Bot API for channel broadcast</li>

            <p>A pipeline that turns raw ideas into published, cross-platform content:</p>
            <p><strong>Input:</strong> A bullet-point outline in a Google Sheet or Notion doc.</p>
            <p><strong>Pipeline steps:</strong></p>
            <ol>
                <li><strong>Expand:</strong> AI agent reads the outline and generates a full blog post draft</li>
                <li><strong>Review:</strong> Draft goes to a "Review" column in Google Sheets for human approval</li>

                <li><strong>Distribute:</strong> Posts tweet thread via Twitter API, LinkedIn article via LinkedIn API, newsletter excerpt via Resend, and shares links in Telegram/Discord communities</li>
                <li><strong>Track:</strong> After 48 hours, pull engagement metrics from each platform and update the content performance spreadsheet</li>

    <AccordionItem value="item-4">
        <AccordionTrigger className="text-lg text-left">Building Resilient Pipelines</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Production pipelines must handle failures gracefully. Key patterns:</p>
            <p><strong>1. Retry with Exponential Backoff:</strong></p>
            <pre><code>{`async function fetchWithRetry(url, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      const response = await fetch(url);
      if (response.status === 429) { // Rate limited
        const delay = Math.pow(2, attempt) * 1000; // 1s, 2s, 4s
        await new Promise(r => setTimeout(r, delay));
        continue;
      }
      return await response.json();
    } catch (error) {
      if (attempt === maxRetries - 1) throw error;
      await new Promise(r => setTimeout(r, 1000 * (attempt + 1)));
    }
  }
}`}</code></pre>
            <p><strong>2. Dead Letter Queue:</strong> When a step fails after all retries, don't lose the data. Store failed items in a separate queue (a database table, a Google Sheet tab) for manual review or later retry.</p>
            <p><strong>3. Idempotency:</strong> Design each step so running it twice with the same input produces the same result. Use unique IDs and "upsert" operations instead of blind inserts.</p>
            <p><strong>4. Circuit Breaker:</strong> If an API is consistently failing, stop calling it and alert the team instead of burning through rate limits with retries.</p>
        </AccordionContent>
    </AccordionItem>
