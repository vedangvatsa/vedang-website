An automation that only works when you're watching isn't really an automation. Production-grade pipelines run reliably 24/7, recover from errors, alert you only when needed, and cost predictably.

The transition from development to production is where most automation projects fail. A pipeline that works perfectly in testing breaks in production because an API rate-limits you, a cron job overlaps with the previous run, or an edge case in the data causes a silent failure that compounds over days. This module covers the operational infrastructure that prevents these failures: scheduling strategies, heartbeat monitoring, structured error handling, cost tracking, and the observability patterns that let you sleep at night while your automations run.

name: Daily Job Broadcast
on:
  schedule:
    - cron: '0 8 * * *'  # Every day at 8am UTC
  workflow_dispatch: {}   # Manual trigger for testing

jobs:
  broadcast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: node scripts/broadcast-jobs.mjs
        env:
          SUPABASE_URL: \${{ secrets.SUPABASE_URL }}
          TELEGRAM_BOT_TOKEN: \${{ secrets.TELEGRAM_BOT_TOKEN }}`}</code></pre>
            <p><strong>2. Vercel Cron Jobs</strong>. For Next.js apps, add cron triggers to API routes:</p>
            <pre><code>{`// vercel.json
{
  "crons": [{
    "path": "/api/cron/broadcast",
    "schedule": "0 8 * * *"
  }]
}`}</code></pre>
            <p><strong>3. n8n / Make Schedules</strong>. Built-in scheduling in no-code tools. Best for non-technical teams.</p>
            <p><strong>4. Railway / Render Cron</strong>. For standalone scripts that need more compute time than GitHub Actions allows (6 hours max).</p>

        <AccordionTrigger className="text-lg text-left">Monitoring & Alerting</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>You need to know when automations fail. before your users notice.</p>
            <p><strong>Lightweight monitoring stack:</strong></p>
            <pre><code>{`// At the end of every pipeline run, send a heartbeat
async function reportPipelineStatus(pipeline, results) {
  const status = {
    pipeline,
    timestamp: new Date().toISOString(),
    success: results.successes,
    failures: results.failures,
    duration: results.durationMs,
  };

  // 1. Log to console (GitHub Actions captures this)
  console.log(JSON.stringify(status));

  // 2. Update monitoring spreadsheet via MCP
  await updateSheet("Automation Log", status);

  // 3. Alert on failure
  if (results.failures > 0) {
    await sendSlackAlert(\`Pipeline \${pipeline} had \${results.failures} failures\`);
  }

  // 4. Heartbeat. external monitoring tools (UptimeRobot, Better Uptime)
  //    will alert if the heartbeat stops arriving
  await fetch(\`https://heartbeat.uptimerobot.com/\${HEARTBEAT_ID}\`);
}`}</code></pre>
            <p><strong>What to monitor:</strong></p>
            <ul>
                <li>Did the pipeline run on time? (Missing heartbeat = alert)</li>
                <li>How many items were processed vs. failed?</li>
                <li>How long did it take? (Duration spike = something is wrong)</li>
                <li>What's the error rate trend? (Gradual increase = API change)</li>
            </ul>
        </AccordionContent>

        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>AI-powered automations have running costs. Track and optimize them:</p>
            <p><strong>Cost categories:</strong></p>
            <ul>
                <li><strong>AI API calls:</strong> OpenAI, Anthropic. charged per token. Use cheaper models (GPT-4o-mini, Haiku) for simple tasks, expensive models only when quality matters.</li>
                <li><strong>Hosting:</strong> GitHub Actions (2,000 free minutes/month), Vercel (free hobby tier), Railway ($5/month for always-on).</li>
                <li><strong>No-code tools:</strong> Zapier charges per task, Make charges per operation. n8n self-hosted = only hosting costs.</li>
                <li><strong>Database:</strong> Supabase free tier covers most use cases. Neon, PlanetScale also have generous free tiers.</li>
            </ul>
            <p><strong>Cost optimization tips:</strong></p>
            <ul>
                <li>Cache API responses to avoid redundant calls</li>
                <li>Use GPT-4o-mini for classification/formatting, reserve Claude/GPT-4 for complex reasoning</li>
                <li>Batch operations. send 10 messages in one API call instead of 10 separate calls</li>
                <li>Run compute-heavy tasks during off-peak hours</li>
                <li>Set billing alerts on all AI API accounts</li>
            </ul>

        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>As you build more automations, organization becomes critical:</p>
            <p><strong>1. Documentation:</strong> Every automation should have a README with: what it does, what triggers it, what services it connects to, what secrets it needs, and how to test it. Future-you (or your team) will thank you.</p>
            <p><strong>2. Version Control:</strong> Keep all automation code in Git repositories. Use the same workflow you'd use for any software. branches, pull requests, code review. The [Vibe Coding](/vibe-coding) course covers these fundamentals.</p>
            <p><strong>3. Centralized Secrets:</strong> Use a secrets manager (GitHub Secrets, Vercel Environment Variables, 1Password) instead of scattering API keys across config files.</p>
            <p><strong>4. Automation Inventory:</strong> Maintain a spreadsheet or dashboard listing all running automations:</p>
            <div className="grid md:grid-cols-2 gap-4 my-4">
                <div className="p-4 rounded-lg border">
                    <h4 className="font-semibold text-sm mb-2">Track for each automation</h4>
                    <ul className="text-sm text-muted-foreground space-y-1">
                        <li>• Name and description</li>
                        <li>• Schedule (when does it run?)</li>
                        <li>• Dependencies (what APIs/services?)</li>
                        <li>• Owner (who maintains it?)</li>
                        <li>• Last successful run</li>
                        <li>• Monthly cost estimate</li>
                    </ul>
                </div>
                <div className="p-4 rounded-lg border">
                    <h4 className="font-semibold text-sm mb-2">Review quarterly</h4>
                    <ul className="text-sm text-muted-foreground space-y-1">
                        <li>• Is this automation still needed?</li>
                        <li>• Has the cost changed?</li>
                        <li>• Are there error patterns?</li>
                        <li>• Can it be consolidated with another?</li>
                        <li>• Should it be upgraded (rule → AI)?</li>
                    </ul>
                </div>
            </div>

    <AccordionItem value="item-5">
        <AccordionTrigger className="text-lg text-left">AI Discovery: Making Your Automations Findable</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>If your automations produce public-facing content (websites, APIs, data feeds), you should optimize for AI discovery. This means ensuring AI agents and search engines can find, understand, and cite your content.</p>
            <p><strong>Essential discovery files:</strong></p>
            <ul>
                <li><strong><code>llms.txt</code></strong>. A markdown summary of your site designed for LLMs. Describes what your site is, what it offers, and links to key pages. See <a href="https://llmstxt.org" target="_blank">llmstxt.org</a> for the format.</li>
                <li><strong><code>agents.json</code></strong>. A machine-readable file at <code>/.well-known/agents.json</code> that tells autonomous agents what your site can do, what APIs are available, and how to interact.</li>
                <li><strong><code>ai-plugin.json</code></strong>. The OpenAI plugin format. Describes your site's capabilities for ChatGPT and other platforms.</li>
                <li><strong><code>robots.txt</code></strong>. Updated to explicitly welcome AI crawlers: OAI-SearchBot, PerplexityBot, ClaudeBot, Google-Extended.</li>
            </ul>
            <p>Deploy all of these at once with the <a href="https://github.com/vedangvatsa/ai-discovery-standards" target="_blank">AI Discovery Standards</a> framework: <code>npx github:vedangvatsa/ai-discovery-standards</code></p>
            <p><strong>Why this matters for automation:</strong> If your automated content pipeline publishes blog posts, those posts need to be discoverable by Perplexity, ChatGPT Search, and Google AI Overviews. Without proper discovery files, your automated content is invisible to the fastest-growing search channels.</p>
        </AccordionContent>

    <AccordionItem value="item-6">
        <AccordionTrigger className="text-lg text-left">Security Best Practices for Automated Systems</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Automated systems have larger attack surfaces than manual processes. Follow these security practices:</p>
            <p><strong>1. Secret management:</strong></p>
            <ul>
                <li>Never commit API keys, tokens, or passwords to Git. Ever. Use <code>.env</code> files locally and GitHub Secrets / Vercel Environment Variables in production.</li>
                <li>Rotate API keys quarterly. Set calendar reminders.</li>
                <li>Use separate API keys for development and production.</li>
            </ul>
            <p><strong>2. Principle of least privilege:</strong></p>
            <ul>
                <li>Your automation should only have the permissions it needs. If it only reads from a database, don't give it write access.</li>
                <li>Use read-only API keys where possible. Create separate keys for different pipelines.</li>
                <li>Google Sheets MCP: share sheets with the service account as "Viewer" unless the pipeline needs to write.</li>
            </ul>
            <p><strong>3. Input validation:</strong></p>
            <ul>
                <li>If your automation processes external data (webhooks, scraped content, user input), validate and sanitize everything before acting on it.</li>
                <li>Never pass raw external data into system prompts or database queries without sanitization.</li>
            </ul>
            <p><strong>4. Audit logging:</strong></p>
            <ul>
                <li>Log every action your automation takes: what it did, when, what data it processed, what the outcome was.</li>
                <li>Store logs in a separate location from the automation itself (Google Sheets tab, a log database, or a file in a protected repo).</li>
                <li>Review logs weekly. Look for unexpected patterns: unusual volumes, failed auth attempts, or data anomalies.</li>
            </ul>
        </AccordionContent>
