---
title: "Transports & Message Format"
description: "MCP uses JSON-RPC 2.0 over multiple transport layers. This module covers how messages flow between clients and servers. from local stdio to remote HTTP."
---

<TransportDiagram />

MCP is transport-agnostic: the same protocol works whether the server runs on your local machine or across the internet. Understanding the transport layer is essential because it determines how you deploy, test, and scale your servers.

<Accordion type="single" collapsible className="w-full mt-4">
    <AccordionItem value="item-1">
        <AccordionTrigger className="text-lg text-left">JSON-RPC 2.0: The Message Format</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Every MCP message is a JSON-RPC 2.0 message. This is a simple, well-established standard for remote procedure calls. There are three types:</p>
            <p><strong>1. Request</strong> (client → server, expects a response):</p>
            <pre><code>{`{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "search_database",
    "arguments": { "query": "SELECT * FROM users LIMIT 5" }
  }
}`}</code></pre>
            <p><strong>2. Response</strong> (server → client):</p>
            <pre><code>{`{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [{ "type": "text", "text": "Found 5 users..." }]
  }
}`}</code></pre>
            <p><strong>3. Notification</strong> (either direction, no response expected):</p>
            <pre><code>{`{
  "jsonrpc": "2.0",
  "method": "notifications/progress",
  "params": { "progressToken": "abc", "progress": 50, "total": 100 }
}`}</code></pre>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-2">
        <AccordionTrigger className="text-lg text-left">stdio Transport: Local Development</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p><strong>stdio</strong> (standard input/output) is the simplest transport. The host launches the MCP server as a child process and communicates through stdin/stdout pipes. Each message is a line of JSON.</p>
            <p><strong>How it works:</strong></p>
            <ul>
                <li>The host spawns the server process (e.g., <code>node my-server.js</code>)</li>
                <li>Messages go to the server via stdin</li>
                <li>Responses come back via stdout</li>
                <li>Errors and logs go to stderr (never stdout. that would corrupt the protocol)</li>
            </ul>
            <p><strong>Best for:</strong> Local development, CLI tools, desktop applications like Claude Desktop. Zero network configuration. The server runs on the same machine as the host.</p>
            <p><strong>Limitation:</strong> Can't work over a network. The server must be a local process.</p>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-3">
        <AccordionTrigger className="text-lg text-left">SSE Transport: Remote Servers</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p><strong>Server-Sent Events (SSE)</strong> enables MCP over HTTP, allowing remote servers. The client sends requests via HTTP POST, and the server streams responses back via SSE.</p>
            <p><strong>How it works:</strong></p>
            <ul>
                <li>Server exposes two HTTP endpoints: <code>/sse</code> (for the event stream) and <code>/messages</code> (for receiving requests)</li>
                <li>Client opens a persistent SSE connection to <code>/sse</code></li>
                <li>Client sends requests via POST to <code>/messages</code></li>
                <li>Server responds via the SSE stream</li>
            </ul>
            <p><strong>Best for:</strong> Remote/cloud-hosted MCP servers, multi-user environments, servers behind firewalls with HTTP access.</p>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-4">
        <AccordionTrigger className="text-lg text-left">Streamable HTTP: The Future</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p><strong>Streamable HTTP</strong> is the newest transport, designed to be the best of both worlds. It uses a single HTTP endpoint and supports both request-response and streaming patterns.</p>
            <p><strong>Key advantages:</strong></p>
            <ul>
                <li><strong>Stateless by default:</strong> No persistent connections needed. Works naturally with serverless platforms (Vercel, AWS Lambda, Cloudflare Workers).</li>
                <li><strong>Streaming when needed:</strong> Can upgrade to SSE for long-running operations, then drop back to stateless.</li>
                <li><strong>Simpler infrastructure:</strong> One endpoint instead of two. Standard HTTP semantics. Works with all existing HTTP infrastructure (load balancers, CDNs, proxies).</li>
            </ul>
            <p><strong>Best for:</strong> Production deployments, serverless environments, APIs that need to scale.</p>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-5">
        <AccordionTrigger className="text-lg text-left">Choosing the Right Transport</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Use this decision tree:</p>
            <p><strong>Is your server running on the same machine as the client?</strong></p>
            <ul>
                <li><strong>Yes → Use stdio.</strong> Zero configuration. No network setup. The host spawns your server as a child process. This is the default for local development and Claude Desktop.</li>
                <li><strong>No → Is it a serverless environment (Vercel, Lambda, Cloudflare Workers)?</strong></li>
            </ul>
            <p>If serverless:</p>
            <ul>
                <li><strong>Yes → Use Streamable HTTP.</strong> Stateless by default. Works naturally with serverless architectures. One endpoint. Standard HTTP.</li>
                <li><strong>No → Use SSE or Streamable HTTP.</strong> SSE is more established and widely supported. Streamable HTTP is newer but better for production.</li>
            </ul>
            <p><strong>Common deployment patterns:</strong></p>
            <ul>
                <li><strong>Local dev + Claude Desktop:</strong> stdio (always)</li>
                <li><strong>Shared team server on a VPS:</strong> SSE or Streamable HTTP</li>
                <li><strong>Public MCP server (npm package):</strong> stdio (user runs locally)</li>
                <li><strong>SaaS product offering MCP:</strong> Streamable HTTP behind authentication</li>
            </ul>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-6">
        <AccordionTrigger className="text-lg text-left">Debugging Transport Issues</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Transport issues are the most common reason MCP connections fail. Here's how to diagnose them:</p>
            <p><strong>1. "Server not starting" with stdio:</strong></p>
            <ul>
                <li>Run the command manually: <code>npx tsx src/index.ts</code>. Does it even start?</li>
                <li>Check for compilation errors. TypeScript errors will crash before MCP initializes.</li>
                <li>Verify the absolute path in your config. Relative paths often break.</li>
            </ul>
            <p><strong>2. "Connected but no tools showing":</strong></p>
            <ul>
                <li>Your tools might be registered after the transport connects. Register tools before calling <code>server.connect(transport)</code>.</li>
                <li>Use the MCP Inspector to verify tools are listed: <code>npx @modelcontextprotocol/inspector npx tsx src/index.ts</code></li>
            </ul>
            <p><strong>3. "Protocol error / corrupted messages" with stdio:</strong></p>
            <ul>
                <li>You're writing to stdout. Any <code>console.log()</code> in your server corrupts the JSON-RPC stream. Use <code>console.error()</code> instead.</li>
                <li>Dependencies might be logging to stdout. Check all imported libraries.</li>
            </ul>
            <p><strong>4. "Connection refused" with SSE/HTTP:</strong></p>
            <ul>
                <li>Check the port. Is another process using it?</li>
                <li>Check CORS headers if connecting from a browser-based client.</li>
                <li>Verify firewall rules allow the connection.</li>
            </ul>
        </AccordionContent>
    </AccordionItem>
</Accordion>
