Theory is over. In this module, you'll build a working MCP server that exposes a custom tool. By the end, you'll have a server running in Claude Desktop that the AI can call on demand.

<Accordion type="single" collapsible className="w-full mt-4">
    <AccordionItem value="item-1">
        <AccordionTrigger className="text-lg text-left">Step 1: Project Setup</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Create a new project and install the MCP SDK:</p>
            <pre><code>{`mkdir my-mcp-server && cd my-mcp-server
npm init -y
npm install @modelcontextprotocol/sdk zod
npm install -D typescript @types/node tsx`}</code></pre>
            <p>Create a <code>tsconfig.json</code>:</p>
            <pre><code>{`{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "outDir": "./dist",
    "strict": true,
    "esModuleInterop": true
  },
  "include": ["src/**/*"]
}`}</code></pre>
            <p><strong>Key dependencies:</strong></p>
            <ul>
                <li><code>@modelcontextprotocol/sdk</code>. The official MCP SDK. Handles all protocol details.</li>
                <li><code>zod</code>. Schema validation library. Used to define tool parameters with type safety.</li>
                <li><code>tsx</code>. TypeScript runner for development. Run without compiling.</li>
            </ul>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-2">
        <AccordionTrigger className="text-lg text-left">Step 2: Write the Server</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Create <code>src/index.ts</code> with a minimal server that has one tool. a word counter:</p>
            <pre><code>{`import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

// 1. Create the server
const server = new McpServer({
  name: "word-counter",
  version: "1.0.0",
});

// 2. Register a tool
server.tool(
  "count_words",                    // Tool name
  "Count the words in a text",       // Description (LLM reads this!)
  { text: z.string() },              // Parameter schema
  async ({ text }) => {              // Handler function
    const count = text.split(/\\s+/).filter(Boolean).length;
    return {
      content: [{
        type: "text",
        text: \`The text contains \${count} words.\`,
      }],
    };
  }
);

// 3. Connect transport and start
const transport = new StdioServerTransport();
await server.connect(transport);`}</code></pre>
            <p><strong>Every tool has four parts:</strong></p>
            <ol>
                <li><strong>Name:</strong> A unique identifier the LLM uses to call the tool.</li>
                <li><strong>Description:</strong> Natural language explanation. The LLM reads this to decide when to use the tool. Write it clearly.</li>
                <li><strong>Schema:</strong> A Zod schema defining the expected parameters. This provides type safety and automatic validation.</li>
                <li><strong>Handler:</strong> An async function that does the actual work and returns a result.</li>
            </ol>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-3">
        <AccordionTrigger className="text-lg text-left">Step 3: Test with MCP Inspector</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Before connecting to a real client, test your server with the <strong>MCP Inspector</strong>. a web-based debugging tool:</p>
            <pre><code>{`npx @modelcontextprotocol/inspector npx tsx src/index.ts`}</code></pre>
            <p>This opens a browser UI where you can:</p>
            <ul>
                <li>See all registered tools, resources, and prompts</li>
                <li>Manually call tools with test parameters</li>
                <li>Inspect the JSON-RPC messages being exchanged</li>
                <li>Verify your server responds correctly before exposing it to an LLM</li>
            </ul>
            <p>The Inspector is your best friend during development. Use it before every deployment to catch issues early.</p>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-4">
        <AccordionTrigger className="text-lg text-left">Step 4: Connect to Claude Desktop</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>To connect your server to Claude Desktop, edit the configuration file:</p>
            <p><strong>macOS:</strong> <code>~/Library/Application Support/Claude/claude_desktop_config.json</code></p>
            <p><strong>Windows:</strong> <code>%APPDATA%\Claude\claude_desktop_config.json</code></p>
            <pre><code>{`{
  "mcpServers": {
    "word-counter": {
      "command": "npx",
      "args": ["tsx", "/absolute/path/to/src/index.ts"]
    }
  }
}`}</code></pre>
            <p>After saving, restart Claude Desktop. You should see a hammer icon indicating available tools. Now ask Claude: <em>"Count the words in this sentence: The quick brown fox jumps over the lazy dog."</em></p>
            <p>Claude will automatically call your <code>count_words</code> tool and return the result.</p>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-5">
        <AccordionTrigger className="text-lg text-left">Writing Tool Descriptions That Work</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>The tool description is the most underrated part of MCP development. The LLM reads it to decide when and how to use your tool. A bad description means the LLM will use your tool incorrectly or not at all.</p>
            <p><strong>Bad descriptions:</strong></p>
            <ul>
                <li><code>"Search"</code>. Search what? How? What does it return?</li>
                <li><code>"Database tool"</code>. Does it read? Write? Both? What database?</li>
                <li><code>"Helper function for data processing"</code>. Tells the LLM nothing actionable.</li>
            </ul>
            <p><strong>Good descriptions:</strong></p>
            <ul>
                <li><code>"Search for users by name or email in the PostgreSQL database. Returns up to 10 matching records with id, name, email, and created_at."</code></li>
                <li><code>"Send a message to a Slack channel. Requires channel name (without #) and message text. Supports markdown formatting."</code></li>
                <li><code>"Fetch the current stock price for a given ticker symbol. Returns price in USD, change percentage, and trading volume."</code></li>
            </ul>
            <p><strong>The rule:</strong> Describe WHAT the tool does, WHAT inputs it expects, and WHAT it returns. Be specific enough that a human who has never seen your codebase could use it correctly.</p>
            <p><strong>Parameter descriptions matter too:</strong></p>
            <pre><code>{`// Bad: no parameter descriptions
{ query: z.string() }

// Good: every parameter described
{
  query: z.string().describe("SQL SELECT query to execute against the users table"),
  limit: z.number().default(10).describe("Maximum results to return (1-100)"),
}`}</code></pre>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-6">
        <AccordionTrigger className="text-lg text-left">Common First-Server Mistakes</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Every MCP developer hits these issues. Save yourself the debugging time:</p>
            <p><strong>1. Using console.log instead of console.error.</strong> In stdio transport, stdout is reserved for JSON-RPC messages. Any <code>console.log()</code> corrupts the protocol stream. Always use <code>console.error()</code> for debugging.</p>
            <p><strong>2. Forgetting to return the right format.</strong> Every tool must return <code>{'{ content: [{ type: "text", text: "..." }] }'}</code>. Returning a raw string or object will cause a protocol error.</p>
            <p><strong>3. Not handling async errors.</strong> If your tool handler throws an unhandled error, the entire server process may crash. Wrap everything in try/catch:</p>
            <pre><code>{`server.tool("my_tool", "Does something", { input: z.string() },
  async ({ input }) => {
    try {
      const result = await doSomething(input);
      return { content: [{ type: "text", text: result }] };
    } catch (error) {
      return {
        content: [{ type: "text", text: \`Error: \${String(error)}\` }],
        isError: true,
      };
    }
  }
);`}</code></pre>
            <p><strong>4. Using relative paths in config.</strong> Claude Desktop and Cursor resolve paths from their own working directory, not your project. Always use absolute paths in <code>claude_desktop_config.json</code>.</p>
            <p><strong>5. Not restarting the client.</strong> After changing your server code or config, you must restart the MCP client (Claude Desktop, Cursor) for changes to take effect.</p>
        </AccordionContent>
    </AccordionItem>
</Accordion>
