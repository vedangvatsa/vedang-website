Now that you understand the fundamentals, let's build real servers that solve actual problems. This module covers three common patterns: database access, API wrappers, and authenticated services.

<Accordion type="single" collapsible className="w-full mt-4">
    <AccordionItem value="item-1">
        <AccordionTrigger className="text-lg text-left">Pattern 1: Database Server</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>A database MCP server lets AI query your database using natural language. The LLM generates SQL, your server validates and executes it, and returns formatted results.</p>
            <p><strong>Key design decisions:</strong></p>
            <ul>
                <li><strong>Read-only vs. read-write:</strong> Start with read-only. Expose a <code>query_database</code> tool that only runs SELECT statements. Add write tools later with strict validation.</li>
                <li><strong>Schema as a resource:</strong> Expose your database schema as an MCP resource. This gives the LLM context about your tables and columns so it generates correct SQL.</li>
                <li><strong>Query validation:</strong> Always validate queries before execution. Block DROP, DELETE, and ALTER statements unless explicitly intended.</li>
            </ul>
            <pre><code>{`// Expose schema as a resource
server.resource(
  "db://myapp/schema",
  "Database schema for the application",
  async () => ({
    contents: [{
      uri: "db://myapp/schema",
      mimeType: "text/plain",
      text: await getSchemaDescription(),
    }],
  })
);

// Read-only query tool
server.tool(
  "query_database",
  "Execute a read-only SQL query against the database",
  {
    sql: z.string().describe("A SELECT query to execute"),
  },
  async ({ sql }) => {
    // Safety check
    if (!/^\\s*SELECT/i.test(sql)) {
      return {
        content: [{ type: "text", text: "Error: Only SELECT queries are allowed." }],
        isError: true,
      };
    }
    const results = await db.query(sql);
    return {
      content: [{ type: "text", text: JSON.stringify(results, null, 2) }],
    };
  }
);`}</code></pre>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-2">
        <AccordionTrigger className="text-lg text-left">Pattern 2: API Wrapper Server</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Wrapping a REST API as an MCP server is one of the most common and valuable patterns. It lets AI interact with any web service.</p>
            <p><strong>Example: Weather API server</strong></p>
            <pre><code>{`server.tool(
  "get_weather",
  "Get the current weather for a city",
  {
    city: z.string().describe("City name, e.g. 'London'"),
    units: z.enum(["celsius", "fahrenheit"])
      .default("celsius")
      .describe("Temperature units"),
  },
  async ({ city, units }) => {
    const response = await fetch(
      \`https://api.weather.com/v1/current?\` +
      \`city=\${encodeURIComponent(city)}&units=\${units}\`,
      { headers: { "Authorization": \`Bearer \${process.env.WEATHER_API_KEY}\` } }
    );

    if (!response.ok) {
      return {
        content: [{ type: "text", text: \`Weather API error: \${response.status}\` }],
        isError: true,
      };
    }

    const data = await response.json();
    return {
      content: [{
        type: "text",
        text: \`Weather in \${city}: \${data.temp}°, \${data.description}\`,
      }],
    };
  }
);`}</code></pre>
            <p><strong>Best practices for API wrappers:</strong></p>
            <ul>
                <li>Always handle API errors gracefully. return <code>isError: true</code> with a descriptive message</li>
                <li>Use environment variables for API keys. never hardcode secrets</li>
                <li>Add rate limiting to prevent the LLM from making too many API calls</li>
                <li>Format responses for readability. the LLM will process them better</li>
            </ul>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-3">
        <AccordionTrigger className="text-lg text-left">Pattern 3: Multi-Tool Servers</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Real servers typically expose multiple related tools. Group tools logically and use consistent naming conventions:</p>
            <pre><code>{`// A project management MCP server
server.tool("list_projects", "List all active projects", {}, ...);
server.tool("get_project", "Get details of a specific project",
  { id: z.string() }, ...);
server.tool("create_task", "Create a new task in a project",
  { projectId: z.string(), title: z.string(), assignee: z.string().optional() }, ...);
server.tool("update_task_status", "Update the status of a task",
  { taskId: z.string(), status: z.enum(["todo", "in-progress", "done"]) }, ...);

// Expose project list as a resource for context
server.resource("projects://active", "List of active projects", ...);`}</code></pre>
            <p><strong>Naming conventions:</strong></p>
            <ul>
                <li>Use <code>verb_noun</code> format: <code>search_users</code>, <code>create_task</code>, <code>get_project</code></li>
                <li>Be specific: <code>search_users_by_email</code> is better than <code>search</code></li>
                <li>Group related tools with a common prefix: <code>project_list</code>, <code>project_create</code>, <code>project_update</code></li>
            </ul>
        </AccordionContent>
    </AccordionItem>
    <AccordionItem value="item-4">
        <AccordionTrigger className="text-lg text-left">Pattern 4: Authenticated Servers</AccordionTrigger>
        <AccordionContent className="prose dark:prose-invert max-w-none text-base">
            <p>Many real-world MCP servers need authentication. either to access external APIs or to verify the identity of the connecting client.</p>
            <p><strong>API Key Authentication (Server → External API):</strong></p>
            <ul>
                <li>Store API keys in environment variables</li>
                <li>Pass them via the <code>env</code> field in the client configuration</li>
                <li>Never log or expose API keys in tool responses</li>
            </ul>
            <p><strong>OAuth 2.0 (Client → Server):</strong></p>
            <ul>
                <li>For remote MCP servers accessed over HTTP, OAuth 2.0 is the standard authentication mechanism</li>
                <li>The client obtains an access token and includes it in HTTP headers</li>
                <li>The server validates the token before processing requests</li>
            </ul>
            <p><strong>Security checklist:</strong></p>
            <ul>
                <li>✅ All secrets in environment variables, never in code</li>
                <li>✅ Input validation on every tool parameter</li>
                <li>✅ Principle of least privilege. only request permissions you need</li>
                <li>✅ Audit logging for all tool invocations</li>
                <li>✅ Rate limiting to prevent abuse</li>
            </ul>
        </AccordionContent>
    </AccordionItem>
</Accordion>
